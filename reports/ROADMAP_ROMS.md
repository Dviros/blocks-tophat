# BLOCKS "tophat" — Improvements + Custom-Firmware/ROM Feasibility Roadmap

**Device:** BLOCKS modular smartwatch prototype, codename `tophat`, ODM Compal
**SoC:** Qualcomm Snapdragon Wear 2100 (APQ8009w / MSM8909w), quad Cortex-A7, 32-bit only
**OS:** Android 6.0.1 (MMB29M), build `eng.tincho5588.20171123.001812`, test-keys, user build
**RAM/Storage:** 512 MB / 3.6 GB eMMC
**Security state (verified):** SELinux permissive, bootloader unlocked, secure boot fused OFF (Sahara reports "possibly unfused device," accepts unsigned images), no root, stock `adbd` compiled `ALLOW_ADBD_ROOT=0`
**Method:** Static analysis only — `strings`/`grep`/`file`/`python3` against `extracted/system`, `extracted/boot/ramdisk_ro`, `extracted/boot/kernel`; `jadx` decompilation of `CronoBridgeService.apk`; WebSearch/WebFetch for external project verification. No USB device touched, no git operations performed as part of this report.

This document assumes the reader has already read `extracted/SYSTEM_ANALYSIS.md` (architecture, app inventory, module-bus mechanism). This roadmap is action-oriented: what to actually do, in what order, for how much benefit, at what risk.

## 0. New evidence found while producing this roadmap

A few things turned up during this pass that materially change the picture in `SYSTEM_ANALYSIS.md` §8 and the repo's own `README.md`/`BACKUP.md`:

1. **A "rooted" boot image already exists on disk and it is NOT root.** `boot-rooted.img` (repo root, 10.9 MB, valid `ANDROID!` boot image) and `extracted/boot/ramdisk-rooted.gz` are **byte-identical** (confirmed via MD5: `1c7fc24b2f939a4eb637554717bc3639` for both, after unpacking `boot-rooted.img` with `tools/mkbootimg/unpack_bootimg.py` and gunzipping both ramdisks). Diffing `default.prop` between stock and this "rooted" variant shows only a property flip:
   ```
   stock:   ro.secure=1   ro.debuggable=0   ro.adb.secure=1
   rooted:  ro.secure=0   ro.debuggable=1   ro.adb.secure=0
   ```
   `boot-rooted.img`'s repacked cmdline also adds `androidboot.selinux=permissive` (redundant — SELinux is already permissive) and drops nothing else. **No Magisk strings, no `su` binary, no init.rc service changes** are present in this ramdisk — confirmed via `strings boot-rooted.img | grep -i magisk` (zero hits) and `cpio -id` extraction of the ramdisk followed by directory listing. This artifact was produced by exactly the repo's own `tools/patch_ramdisk_prop.py` script, which is a clean, well-written surgical cpio editor — but the repo's own `README.md` already states, correctly and honestly, that **"property-flip alone does not work (user-build adbd)."** This tracks: flipping `ro.debuggable`/`ro.secure` does nothing for `adbd`'s ability to `setuid(0)` a shell, because that gate is compiled into the `adbd` binary itself (`ALLOW_ADBD_ROOT` at build time), not read from a runtime property. **Net: this file is a useful debug-flags boot image (verbose logging, `ro.debuggable=1` unlocks some `dumpsys`/strict-mode paths) but grants no `adb shell` root and should not be described as "rooted" in any follow-on work.** Rename it or regenerate it labeled honestly (e.g. `boot-debuggable.img`) to avoid future confusion.

2. **Kernel version extracted and confirmed:** `Linux version 3.18.24-g6578b69 (rdadmin@vBuild1EZW1) (gcc version 4.8 (GCC)) #1 SMP PREEMPT Tue Sep 13 19:26:51 CST 2016`. Method: the `zImage` at `extracted/boot/kernel` (10.1 MB, confirmed via `file` as "Linux kernel ARM boot executable zImage") embeds a self-decompressing gzip stream; located its magic bytes (`1f 8b 08`) at file offset 16495, decompressed with Python's `gzip` module, and `strings`-grepped the ~17 MB decompressed payload. This pins down three separate facts at once:
   - **Kernel baseline: Linux 3.18** (a CodeAurora/CAF LTS branch — see §2 for source availability).
   - **Build host `vBuild1EZW1`** — "EZW1" directly corroborates the "EZW2" codename already found in the module firmware filenames (`EZW2_CoreHub_00_...bin` etc.) in `SYSTEM_ANALYSIS.md` §6, confirming EZW1→EZW2 was an internal Cronologics/BLOCKS hardware-revision lineage, and that the kernel itself long predates the Nov-2017 Android build date (kernel compiled **Sep 2016**, over a year earlier — normal for BSP-vendor-then-OEM-integration timelines, but means kernel-side fixes have an even larger gap to close than the already-noted 2016-05-01 security patch level).
   - **GCC 4.8** toolchain — legacy but unsurprising for a 2016 Qualcomm BSP; relevant if anyone ever attempts an in-place kernel rebuild (a modern GCC/Clang cross-compiler CAN usually still build a 3.18 kernel; this isn't a hard blocker, see §2).

3. **`modulecom_daemon` has zero Bluetooth/BLE/GATT linkage.** Grepped its full symbol table for `ble|gatt|bluetooth` (case-insensitive) — every hit was either libc++ STL symbols coincidentally matching (`condition_variable`, etc.) or `zmq_proxy_steerable`/CoreHub-driver strings. **The module-bus daemon and any phone-companion BLE channel are two entirely separate subsystems** — this corrects an assumption that might otherwise be made from "modulecom_daemon does IPC, therefore it might also do BLE." It does not. This matters for §5 below: don't waste time looking at `modulecom_daemon` for a companion-app bridge.

4. **The real BLE phone-companion implementation was found and identified precisely.** `extracted/system/priv-app/CronoBridgeService/CronoBridgeService.apk`, decompiled with `jadx 1.5.5`, contains `com.cronologics.bridge.ble.peripheral.BlePeripheralCronoBridge.java` and `com.cronologics.bridge.ble.BleUuid.java` — a complete, cleanly-named BLE **peripheral-mode** (watch-as-GATT-server) implementation using `BluetoothGattServer`/`BluetoothLeAdvertiser` (API 21+, used correctly). Full UUID/constant table recovered (see §5 for the complete evidence and citations) — this is concrete, load-bearing evidence, not a guess.

## 1. Root — Ranked

Ranked by (effort × reliability × reversibility), most-recommended first.

### #1 RECOMMENDED — Magisk-patch the boot image
**Effort: Low-Medium. Reliability: High. Reversibility: Perfect (flash-back from `edl_backup`).**

- **Why it's the right call:** the device is exactly at Magisk's stated floor. Magisk's own repository (`github.com/topjohnwu/Magisk`, verified via WebFetch on its README) states it supports **"devices higher than Android 6.0"** — this device is Android 6.0.1, right at/above that line. Separately, community reporting (Android Police / XDA coverage of the Magisk v26 changelog, WebSearch-verified) says v26 (April 2023) dropped Lollipop (5.0/5.1) support specifically, keeping 6.0 as the new floor — consistent with the primary source. **Net: Android 6.0.1 should be supported by current Magisk**, but this is a documented floor, not a device that's been individually tested by upstream — budget time for surprises.
- **Architecture caveat (flag, not blocker):** this device is `armeabi-v7a` only (`ro.product.cpu.abilist=armeabi-v7a,armeabi`, confirmed in `build.prop`, no arm64-v8a). Magisk's release archive has historically shipped both 32-bit (`magisk32`) and 64-bit (`magisk64`) native binaries and auto-selects at boot based on the device's primary ABI, so a 32-bit-only device installing a 32-bit-capable Magisk binary is architecturally the normal case, not an edge case — **but I could not get independent, current (2024-2026) confirmation from WebSearch that a real armeabi-v7a-only device has been Magisk-rooted recently**, since almost all currently-discussed installs are on arm64 phones. **Mark this unverified — the honest recommendation is to try it (it costs nothing but time, and the artifact is 100% reversible) rather than assume it will "just work."**
- **Procedure (per repo's own `README.md`, which is already correct):**
  1. Copy `edl_backup/boot.bin` to any Android phone running the Magisk app.
  2. In Magisk app: Install → "Select and Patch a File" → point at `boot.bin` → produces `magisk_patched-<ver>.img`.
  3. Copy the patched image back to the Mac.
  4. **Test first, don't flash first:** `fastboot boot magisk_patched.img` (boots from RAM, doesn't touch the `boot` partition) — confirms the watch boots and `adb shell` gets a functioning environment before committing.
  5. Only if the RAM-boot test succeeds: `fastboot flash boot magisk_patched.img && fastboot reboot`.
  6. Install the Magisk Manager APK over `adb install` to manage modules/su grants going forward.
- **Reversibility:** flash `edl_backup/boot.bin` back via `fastboot flash boot` or `edl w boot edl_backup/boot.bin` — the full backup already exists and is verified byte-exact (per `BACKUP.md`).
- **Risk:** the only nonzero-probability failure mode is the watch not booting after flash (unlikely given `fastboot boot` was already tested from RAM first) or Magisk's `magiskinit` process hitting something unexpected in this particular non-standard ramdisk (BLOCKS's `init.qcom.rc`-repurposing trick noted in `SYSTEM_ANALYSIS.md` §5 is unusual but shouldn't confuse magiskinit, which patches generically). Given the `fastboot boot`-before-flash safety net and full partition backup, this is about as low-risk as root gets.

### #2 — Direct `su` binary injection into system/vendor + custom init service
**Effort: Medium. Reliability: Medium-High. Reversibility: Good.**

- Cross-compile or source a static `su` binary for `armeabi-v7a`/ARMv7 Android 6.0 (e.g. from an old Chainfire SuperSU release, or build one against a matching NDK), push it into `/system/xbin/su` in a repacked `system.img`, add a `service` stanza or a boot-time script that `chmod`s it 06755 and fixes SELinux context (`u:object_r:su_exec:s0` — this context **already exists** in the compiled `sepolicy`, since `init.rc`'s stock `service adbd` stanza already references `--root_seclabel=u:r:su:s0`, meaning the `su` domain is present in AOSP's stock 6.0 sepolicy even on this device — confirmed conceptually, not independently re-verified against this exact binary sepolicy here).
- **Why it's #2, not #1:** more manual work than Magisk (must source/build the `su` binary yourself, manually wire SELinux domain transitions, no automatic module/update infrastructure afterward) for no real reliability gain over Magisk, which does the exact same thing but packaged, tested at scale, and reversible via its own uninstall path. Only choose this if Magisk genuinely fails on this device (see #1's architecture caveat) — it's the fallback, not a parallel-first choice.
- **Reversibility:** reflash stock `system.bin`/`boot.bin` from `edl_backup/` — clean since the `su` binary and its trigger are both new files added to the images, nothing stock was modified in place.

### #3 — userdebug-style `adbd` swap (replace the `user`-build `adbd` binary with one compiled `ALLOW_ADBD_ROOT=1`)
**Effort: Medium-High. Reliability: Medium. Reversibility: Good.**

- This is the "real" fix for the actual root cause (`adbd`'s `ALLOW_ADBD_ROOT=0` compile flag, confirmed in the task's device facts) rather than a workaround. Requires either (a) finding/building an MMB29M-era AOSP `adbd` for armeabi-v7a with the flag flipped, which means having a matching AOSP source tree checked out at the right tag (freely available — `android.googlesource.com`, tag `android-6.0.1_r*`) and either cross-compiling just that one binary or a broader build, or (b) hex-patching the existing binary's `ALLOW_ADBD_ROOT` check if it's a simple compares against a build-time constant (feasible but fragile without decompiling `adbd` itself, which was not done as part of this static-analysis pass).
- **Why it's #3, not higher:** this is strictly more engineering effort than Magisk for a device-side outcome that's functionally identical (root shell), and it lacks Magisk's ecosystem (Zygisk, systemless hosts-file, module manager, hide-from-detection). The only reason to prefer it is philosophical (no third-party binary blob) or if you specifically want a `userdebug`-flavored adbd rather than the Magisk daemon model. Not recommended as the primary path here.
- **Reversibility:** swap the `adbd` binary back from `edl_backup`'s `system.bin`/ramdisk extraction.

### #4 — NOT recommended as a root method: property-flip alone (what's already on disk)
**Effort: Trivial (already done). Reliability: Zero for actual root. Reversibility: N/A — it grants nothing to revert.**

- Listed for completeness because it's the thing already sitting in the repo (`boot-rooted.img`/`ramdisk-rooted.gz`). As established in §0.1 above, this does **not** produce a root shell — `ro.debuggable=1`/`ro.secure=0` change what some system services allow (verbose debug output, mock locations, some `dumpsys` verbosity) but do not touch `adbd`'s own hardcoded root gate. **Do not present this as "the device is rooted" in any follow-on documentation** — it should be relabeled as a debug-properties boot image, and the actual root effort should proceed via #1.

**Recommendation: pursue #1 (Magisk-patch) first, budget a fallback session for #2 if Magisk's `magiskinit` chokes on anything unusual in this ramdisk.**

## 2. Custom ROM Feasibility — Honest Verdict

### Is there LineageOS/AOSP support for APQ8009w/MSM8909w?

**No official LineageOS device tree exists for MSM8909w, and none should be expected.** LineageOS has never had an official Wear OS / Android Wear device in its supported-devices list at all (verified: `wiki.lineageos.org/devices/` lists phones/tablets, no wearables; general knowledge plus WebSearch found no counter-example) — LineageOS as a project has simply never targeted the Wear OS product category, independent of chipset. This isn't a MSM8909w-specific gap; it's a "Lineage doesn't do watches" gap.

**Generic (non-Wear) MSM8909 has broad community kernel-fork activity** — this is the encouraging half of the picture. WebSearch confirmed multiple actively-forked `android_kernel_*_msm8909` repositories on GitHub for **phone/tablet** devices on this chipset family: LG (`AndroidKernelDevelopment/android_kernel_lg_msm8909`), Alcatel (`zombah/android_kernel_alcatel_msm8909`), Huawei (`KonstaT/android_kernel_huawei_msm8909`, `Explosion-y6/...`, `DestructoSphere/...`), Lenovo tablets (`bbelos/android_kernel_lenovo_msm8909`, `georgy-komarov/...`). These are all standard-Android-phone MSM8909 (not the `w`-suffix Wear variant), so they carry phone-board device trees (different display panel, no watch PMIC rail config, no ambient-mode display path, different touch/battery/charging drivers) — **useful as kernel-source reference and patch material, not directly flashable or even directly device-tree-portable.**

**The one genuinely relevant lead: Huawei Watch 2 (MSM8909w, same exact silicon as this device).** WebSearch found an active XDA thread — `[ROM+KERNEL] HUAWEI WATCH 2 [LTE-Sawshark]+[BT-Sawfish] (OWDE-180926-001-A1, 22 Jan 20)` — describing custom ROM and kernel work for both Huawei Watch 2 variants, plus a separate `[KERNEL] WearOS/AsteroidOS Simple Custom Kernel` thread mentioning "updates for the Qualcomm MSM8909W driver... patches for Android Oreo including scheduler optimizations and governor updates." **This is the single most relevant external artifact found and should be the first place to look for portable kernel patches** — same SoC, same Wear-specific PMIC/board family. **Caveat: I could not fetch the XDA thread directly (WebFetch returned HTTP 403 — XDA blocks non-browser fetches), so I have not personally confirmed a working public kernel-source *repository URL* from that thread — only its existence and rough contents via WebSearch result snippets.** Treat "Huawei Watch 2 kernel/ROM work exists on XDA" as verified-via-search-snippet, and "there is a clone-able git repo with buildable msm8909w kernel source" as **unverified — go read that XDA thread directly (a human, in a browser) before committing engineering time to it.**

**BLOCKS' own later open-source effort (Project OpenWatch, announced March 2018) is the WRONG SoC** — confirmed via WebSearch (Liliputing and OpenSourceForU coverage). Project OpenWatch explicitly targets **MediaTek MTK6580**-based watches (Zeblaze Thor, Lemfo LES1, Kingwear KW88/98/99), because by 2018 BLOCKS' retail product had switched away from this Qualcomm platform. This corroborates and sharpens what the repo's own `README.md` already correctly flags: **the retail "harmony" BLOCKS watch and any BLOCKS-branded open-source work is MediaTek, not Qualcomm — none of it applies to this prototype.** There is no evidence anywhere (BLOCKS, Cronologics, Compal, or Qualcomm) of MSM8909w-specific kernel or device-tree source ever being published for this exact hardware.

### Is kernel source obtainable?

**Partially — the upstream CAF baseline is obtainable; the BLOCKS/tophat-specific board delta almost certainly is not.**

- The kernel is confirmed **Linux 3.18.24** (extracted directly, see §0.2). Qualcomm's CodeAurora Forum (CAF) published `git://codeaurora.org/kernel/msm.git`, and this is mirrored in multiple places, most durably at **`github.com/android-linux-stable/msm-3.18`** ("CAF 3.18 kernel source with linux-stable merged in", tag `kernel.lnx.3.18.r34-rel", confirmed via WebFetch of that repo — note it was **archived/read-only as of May 28, 2021**, so it's a frozen historical snapshot, not a living upstream, but the source is there and clonable). This gives you the **generic CAF 3.18 baseline** — core scheduler, generic MSM platform glue, common Qualcomm drivers.
- **What CAF's generic tree will NOT give you:** the actual `arch/arm/configs/msm8909w_defconfig`-equivalent, the tophat/Compal board `.dts` (device tree source) describing this exact watch's display panel, touch controller, battery-gauge wiring, PMIC rail map, and — critically — **any of the SPI5.0/st-manager CoreHub wiring that `modulecom_daemon` depends on** (per `SYSTEM_ANALYSIS.md` §4, that wiring isn't even visible in the static `/system` + ramdisk image; it's compiled into the kernel/board file, which was never dumped or published). Board-support-package (BSP) deltas like this are exactly the part vendors keep proprietary and BLOCKS/Compal never published theirs.
- **Verdict on source: the generic Linux/CAF layer is obtainable (unverified-but-plausible that `android-linux-stable/msm-3.18` or a similar CAF mirror contains an `msm8909`-family defconfig — this needs someone to actually clone and `ls arch/arm/configs/` to confirm, which was not done here since it requires a full git clone, out of scope for this static-analysis pass). The board-specific delta (tophat's own `.dts`, defconfig, and CoreHub SPI driver, if any exists as an actual kernel module rather than pure userspace) was never published by BLOCKS/Compal/Cronologics and there is no evidence it exists publicly anywhere.**

### Full ROM port vs. modify-the-stock-image-only — the honest verdict

**Modify-the-stock-image-only. A full ground-up AOSP/LineageOS ROM port is not realistic, and recommending otherwise would be dishonest about the effort required.**

Reasoning, stacked:
1. No LineageOS Wear OS precedent exists at all (category gap, not just chipset gap).
2. No published tophat/BLOCKS board support package (device tree, defconfig, or proprietary blobs list) exists anywhere that WebSearch could find.
3. The one same-silicon device with real community ROM work (Huawei Watch 2) has a **different board** — different display, different battery/PMIC wiring, no CoreHub/SPI5.0 module bus — so even a successful Huawei Watch 2 device-tree pull would need the display panel driver, touch driver, and battery-gauge driver entirely re-authored for tophat's actual hardware, which was never characterized in this static analysis (no schematics, no bring-up guide — only what's inferable from the Android-side `/system` dump).
4. **Building a new kernel from the CAF 3.18 baseline without the tophat board file means you'd be reverse-engineering the board wiring from scratch** (probing GPIOs, guessing PMIC rails, trial-and-erroring the display timing) — this is a multi-month bring-up project for one person, not a weekend port, even with the existing kernel as a starting skeleton.
5. **The stock kernel already boots and drives all this device's actual hardware correctly** (it's a working, if old, Android 6.0.1 image) — so the ROI of "keep the stock kernel + BSP, modify only the Android userspace/system image on top" is enormously higher than "replace the kernel," for a fraction of the effort.

**Recommended path:** treat the **kernel + board support package as fixed/frozen** (accept it stays at 3.18.24, 2016-05-01 patch level) and invest all ROM-adjacent effort in **modifying `system.img` in place**: debloat (§3), harden what SELinux/permissions can be hardened without new kernel features (§4), and optionally reflash a **newer AOSP userspace built against the SAME kernel ABI** only if someone is willing to accept the real risk of HAL/vendor-blob mismatches — this last option (e.g., trying to run a much newer Android userspace against this exact kernel) was not evaluated in depth here because it's a large speculative undertaking; flagging it as a "maybe, with major caveats" rather than a recommended path.

**Total safe, low-risk, immediate debloat (excluding the RIL stack, which needs the one verification step above): roughly 700 KB of dead binaries + ~41.5 MB reclaimable via `strip`.** Small in absolute eMMC terms (the partition has room), but each MMI/diag/sample-auth binary removed is one less thing running as an unaudited, permissive-SELinux, potentially-root-adjacent process on a device with essentially no MAC confinement (`SYSTEM_ANALYSIS.md` §7).

## 4. Security Modernization Without Kernel Source

Everything here is scoped to **userspace-only changes** (SELinux policy, `init.rc` edits, permission/ownership changes, Magisk modules) that don't require rebuilding or even having the kernel source, since §2 concluded kernel-source-level work isn't realistic for the board-specific parts.

### 4.1 Fix the `chmod 666` on the CoreHub SPI node (highest-leverage single fix)
- **Current state (confirmed, `SYSTEM_ANALYSIS.md` §4):** `init.rc` line ~469-471 does `chmod 666 /sys/bus/spi/devices/spi5.0/st-manager`, world-writable, with a comment naming "Corehub"/"IPC server". This is the only place module-bus device permissions are set anywhere in the ramdisk.
- **Fix:** replace with a dedicated group (e.g., create a `corehub` AID/group) and `chown root:corehub` + `chmod 660` instead of `666`. Add a matching SELinux type (`corehub_device`) and a policy rule allowing only the `modulecom_daemon_exec` domain (which **already exists** — confirmed via `strings sepolicy | grep -i modulecom`, three domains present: `modulecom_daemon`, `modulecom_daemon_exec`, `modulecom_daemon_tmpfs`) read/write access. This closes the "any app on a permissive-SELinux, world-writable-sysfs box can talk raw SPI to CoreHub" hole precisely as `SYSTEM_ANALYSIS.md` §8 recommends, and — critically — **this is 100% userspace/ramdisk-level work, no kernel rebuild needed**, since sepolicy and `init.rc` are both in the ramdisk/ext4 image, not compiled into the kernel binary.
- **Caveat:** flipping SELinux from permissive to enforcing (a natural next step once policy is tightened) is higher-risk and should be done incrementally — enforce just the new `corehub_device` rule first (permissive **domain-specific** dontaudit/neverallow testing is possible even with the global mode still permissive, via `setenforce`-adjacent per-domain testing), rather than flipping the whole device to global enforcing mode in one step, since permissive-vs-enforcing across the *entire* rest of a device this old and undocumented is a much bigger, riskier undertaking (very likely to break something nobody has characterized).

### 4.2 SELinux hardening of `modulecom_daemon` itself
- It currently runs as **root:root**, class `main`, with only an exec-domain label and **no other confinement** (confirmed: `service_contexts` has no entry for it at all, since it's not a Binder service — it never even registers with `ServiceManager`). A minimal, achievable hardening pass:
  - Move it off `root:root` to a dedicated `corehub` UID/GID if it doesn't genuinely need root for anything beyond opening the SPI node (which, once 4.1's group-based permission model is in place, it won't) — the daemon can then run privilege-dropped after opening the (now group-gated, not world-writable) device node.
  - Write actual SELinux `allow` rules scoping exactly what `modulecom_daemon_t` can touch (currently it's unconfined beyond the bare exec-domain transition — the compiled `sepolicy` has no per-permission rules for this domain per the "only an exec-domain label" finding in `SYSTEM_ANALYSIS.md` §3). This is genuine new-policy-writing work (using `audit2allow`-style workflows against real permissive-mode denial logs collected from actual device use) — moderate effort, no kernel source needed since sepolicy compilation only needs the `sepolicy` tooling + policy source, not kernel headers.
- **This is the single most valuable non-kernel security investment**, because it directly closes the exact vulnerability `SYSTEM_ANALYSIS.md` §7 calls "the single largest untrusted-input attack surface on the device" (a root C++ process parsing binary messages from physically-swappable external hardware).

### 4.3 Magisk-module-based hardening (once root, per §1, is in place)
- **DenyList / hide-root-from-apps:** not very relevant here (no Play Integrity / SafetyNet-dependent apps will ever run on this SDK-23, 32-bit-only, offline device — moot for this use case, unlike a daily-driver phone).
- **Systemless `hosts` / systemless `/system` overlay via Magisk modules:** lets you stage the §3 debloat and §4.1/4.2 sepolicy/permission changes as a **reversible Magisk module** instead of permanently repacking `system.img` — meaning you can `adb reboot recovery`-toggle a module on/off or uninstall it cleanly, rather than needing a full re-flash-from-backup to undo an experiment. **Recommend doing the debloat and SELinux work as Magisk modules first, and only "bake in" to a repacked `system.img` once you're confident in the final state** — this meaningfully de-risks the whole §3/§4 effort.
- **Disabling unused radios (RIL, ANT+) via Magisk module `service.sh`/`post-fs-data.sh` scripts** (e.g., `stop rild` at boot, mask the ANT+ HAL) is a fully reversible, no-kernel, no-repack way to test the §3 RIL/ANT+ debloat hypothesis *before* committing to physically removing those binaries — **this is actually the recommended sequencing**: mask-and-observe via Magisk boot scripts first, remove-from-image only after confirming nothing breaks over a real usage period.

### 4.4 What's realistically NOT achievable without kernel source
- Any actual **kernel CVE backport** for MSM8909/Qualcomm-specific driver bugs (as `SYSTEM_ANALYSIS.md` §8 item 5 optimistically floats) requires the kernel source tree, which per §2 above is only partially available (generic CAF 3.18 baseline yes, tophat board delta no). Realistically: you could pull generic upstream 3.18-branch Linux CVE fixes from the CAF mirror if the vulnerable code path is in shared/generic code (scheduler, generic VFS, generic networking) — genuinely achievable — but MSM-specific driver CVEs (Qualcomm's own camera/GPU/modem driver code, historically a large fraction of high-severity Android kernel CVEs in this era) are **not** fixable without either the real board-specific source or a full binary-patching/hex-editing effort against the compiled kernel image, which is a much higher-skill, higher-risk undertaking not recommended as a primary path.
- **Verified boot / dm-verity re-enablement as a real security boundary:** the fstab already has `verify` wired up for `/system` (confirmed: `fstab.qcom` line reads `/dev/block/bootdevice/by-name/system /system ext4 ro,barrier=1 wait,verify`) and a `verity_key` exists in the ramdisk — but this is **entirely moot** on an unlocked-bootloader, fused-secure-boot-OFF, test-keys device: anyone can re-sign images with their own key or simply not care about verity failures, since there's no hardware root of trust enforcing it. Don't invest effort "re-enabling" verified boot here — it cannot provide real security on this specific unit's fuse configuration, full stop.

### 5.2 GadgetBridge feasibility

**Current state: GadgetBridge does not support BLOCKS/Cronologics devices** (confirmed via WebFetch of `gadgetbridge.org/gadgets/`: full alphabetical vendor list checked, no "Blocks," "Cronologics," or generic modular-smartwatch entry; ~500 supported gadget models from 72 vendors, none matching).

**Could it be added? Yes, and the UUID table above makes this a bounded, scoped task rather than a from-scratch reverse-engineering project:**
- GadgetBridge's own architecture (per its public "New Device Tutorial"/development docs, WebSearch-confirmed) is built around per-device `DeviceSupport` subclasses that talk to specific GATT service/characteristic UUIDs — exactly the shape of integration this watch needs, and exactly the data now available in §5.1's table.
- **Concretely scoped next step (not done here — this is future work, correctly out of scope for a static-analysis pass): write a new GadgetBridge `DeviceSupport` implementation that connects to `CRONO_SERVICE` (0xFE5A), writes requests to `CRONO_CONTROL_POINT`, and subscribes to notifications on `CRONO_DATA_SOURCE`.** The actual wire-protocol *format* of what goes over those two characteristics (message framing, command IDs, etc.) was **not** reverse-engineered in this pass — that would require either deeper `BlePeripheralCronoBridge.java` decompilation (the `.java` file exists and is readable, this pass only pulled the UUID constants from the adjacent `BleUuid.java`) or live BLE sniffing against a real phone-companion pairing session. **This is a well-scoped, medium-effort follow-on task, not a blocker** — the hard part (finding the actual service) is done.
- Note also that since this watch is a peripheral/GATT-server (not a central), a GadgetBridge integration would work the same directional way GadgetBridge already works with everything else (phone-as-central, watch-as-peripheral) — no unusual role-reversal engineering needed.

### 5.3 Reviving the original BLOCKS phone-companion app
- Not evaluated in this pass — the companion Android/iOS app itself (the phone-side app that talks to `CronoBridgeService`) is not part of this device's firmware image and was not analyzed; its availability (an old APK on APKMirror/Internet Archive, or the iOS App Store listing possibly still cached somewhere) was not searched for as part of this task. **Flagging as a concrete, cheap next research step** (a single WebSearch for archived BLOCKS companion app APKs) rather than a firmware-analysis finding.

### 5.4 A fresh, from-scratch BLE bridge (alternative to GadgetBridge integration)
- Given the clean, standards-adjacent GATT design already in place (custom service + control-point/data-source pair, standard CCCD usage), a **minimal custom Android/phone-side app or even a `bleak`(Python)/`noble`(Node) script could connect directly** to `CRONO_SERVICE` without needing GadgetBridge's full device-support framework at all — genuinely the lowest-effort "get *some* companion functionality back" path, at the cost of not getting GadgetBridge's existing notification-formatting/calendar/weather sync infrastructure for free. Good weekend-project scope; §5.2's GadgetBridge path is the better investment if the goal is long-term maintainability.
